Algorithms

In computer science an algorithm is a set of coding instructions that is used to solve a specific problem. On the algorithms contained here are for solving the issue of searching and sorting sets of data. Whether the data is as simple as numbers or something more advanced i.e sorting objects via an id value the approach is more or less the same. In each of the specific algorithms we will show how it works and how to implement it. (In this area the only algorithm that is fleshed out is the merge sort algorithm. All others have a page but contain blank data and greeked layout text.)




Comparisons

This allows us to compare one algorithm for example merge sort to another algorithm i.e. insertion sort, it will show us how their complexity relates to one another and how they handle larger and larger file sets, or larger arrays in the case of elements that we wish to sort. Complexity table

Complexity in relation to algorithms is an important part for us to consider when deciding which algorithm to use. Here we have a table of the algorithms mentioned in this site and others that are known of and potentially usefull. Inside the table we show the algorithm name, its time complexity in best, average and worst case scenarios, and also its space complexity. When picking an algorithm it is useful that we know our hardware limitations and the amount of space we can use so that we can make an informed decision of the algorithm that is best for us.

Merge sort

Insertion sort


Space

Above we see the space complexity for merge sort and insertion sort. Merge sort here as is quit apparant has a much higher use of space i.e. order(n) this is due to the fact that for every element we basically create its own array in a sense using the same amount of space of the original array twice over to get the desired sorting result. For insertion sort it uses order(1) which is the most efficient possible amount of space we can use, i.e. it uses the space of the original stored data and no more as it sorts within the original array itself. Meaning if we want a consitant result and have no care of space requirements then merge sort is ideal however on the flip side if we have very limited space however care little of consistant sorting times then insertion sort is ideal. About complexity

Ok so lets talk complexity, when we rate from great to awful this can be somewhat misleading, we are using the great green to point out that an algorithm is using an order that is so small in comparison to say exponential growth which we see as highlighted in red with the awful tag. The reason for it is anything that is exponitial in growth we generally want to avoid if at all possible. Thats not to say that we wont have to use it there maybe times with indeed we do need to use an algorithm with some type of exponential growth. Also when we signify ordre (n log(n)) as being bad this is say in comparison to order(1) and in fact algorithms with this order i.e. mergesort and quick sort are actually quite often used due to their better consistancy. Obviously logarithmic algorithms i.e. log(n), or algorithms with order(n), or order(n+k) are preffered we also have to be very aware of where we are using them and what specifically we are using them for. At all times our purpose and hardware make big considerations when deciding on an algorithm so dont discount one above that is cited as being bad or awful, as it may well be the one you need.

Merge sort

Above we see two demonstrations of how merge sort works, by splitting each part in half and consistantly doing this then reforming as shown above. We see divide and conquer working, the space complexity is order(n) due to the fact we have to dismantle and create new mini arrays when splitting it up. The best, average and worst case time complexity for merge sort is always (n log(n)) which along side heapsort makes it one of the most consistant algorithms for time to completion which often makes merge sort desireable.
Underneath we see one version of code implementation which is shown in java. Regarding code implementation

Ok lets break the code down and explain exactly how it works. If we move to the main function first at line 80 onward, we see we have an array of 8 elements, the included numbers are in-material, then we create a new mergesort object. First we print the result fo the array as is before applying merge sort, which simply outputs the array. Next we enter the array into the merge_sort function, it takes the array, start element 0, and end element 7. Next onto the merge_sort function, it checks if l is less than r which at this point of course it will be i.e. 0 is less than 7. Next it gets the mid point of the array, and we add the left half and the right half back into merge_sort again. This continues until the left element is not less than the right i.e. there is only one element.


Finally we begin merging the two halves as demonstrated in the image and video at the top of this page. We compare the internal numbers and moving their values consistantly until the whole array is reconstructed in the correct order. Finally the array is again printed after merging and we get a nicely sorting array in for this case an almost instant result.
